# Starshine Geo 0.2.0

Starshine Geo 0.2.0 is the first interoperability and reproducibility release of the public project.
Every implementation, fixture, document, and validation record in this release is maintained in the
public repository and uses public or self-created data only.

## Highlights

- operator-specific JSON Schema contracts for buffer, dissolve, and point-within-polygon summary;
- whole-workflow structural, input, parameter, output, and CRS preflight diagnostics;
- `starshine validate` for checking workflows without reading spatial data or executing operators;
- deterministic, path-free reproducibility manifests with sensitive-value redaction;
- optional GeoPackage read/write support with explicit layers, CRS preservation, and overwrite
  guards;
- real GeoPackage round-trip tests using self-created features in an isolated optional-dependency
  CI job;
- package builds, metadata checks, archive inspection, and public-boundary auditing in CI.

## Compatibility

- Python 3.10, 3.11, and 3.12 are tested for the base package;
- the optional GeoPackage stack is tested on Python 3.11;
- GeoJSON remains the canonical in-memory contract;
- the public operator registry remains closed to dynamic imports and `eval`.

## Upgrade notes from 0.1.0

Workflow validation is intentionally stricter. Unknown workflow, step, input, and parameter fields
are rejected before execution. Buffer steps must declare positive `distance`, a parseable
`source_crs`, and a projected `work_crs`. Consumers that previously relied on operator-time Python
errors should use the structured diagnostic envelope instead.

The runtime version is now read from installed package metadata. `starshine --version`, manifests,
wheel metadata, the changelog, and citation metadata are checked for consistency.

## Public data and source boundary

This release contains no private database, proprietary dataset, textbook material, OCR output,
machine-specific path, internal deployment configuration, or unreleased private source module.
Synthetic workflow and GeoPackage fixtures are created directly in this repository. The CI public
repository audit enforces the tracked-file boundary on every pull request.

## Known limitations

- the public operator set remains intentionally small;
- GeoPackage support is optional and does not provide database discovery;
- no PostGIS adapter, web UI, or private research module is included;
- performance benchmark fixtures remain planned for the 0.2.x maintenance line.

## Validation

The release candidate must pass the Python 3.10–3.12 test matrix, the optional GeoPackage job,
Ruff, Pytest, external JSON Schema fixture validation, the repository boundary audit, wheel/sdist
builds, Twine metadata checks, and archive inspection before tagging.
